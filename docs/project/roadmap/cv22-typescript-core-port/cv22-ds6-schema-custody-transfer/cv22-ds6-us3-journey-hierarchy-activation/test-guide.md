# Test Guide — CV22.DS6.US3 — Journey Hierarchy Activation & Migrate-on-Open

## Test Matrix (acceptance → regime)

| ID | Behavior | Regime | Where |
|----|----------|--------|-------|
| A1 | Legacy DB (no 017) → backup + `017` applied + column backfilled + ordering unchanged | Real-legacy-DB-copy | `ts/parity` harness (redacted) |

## Automated Validation

- **Golden parity (CI, regenerate = no-op gate):**
  - Dual-read: same JSON corpus renders identical `list_journey_options` whether the
    `parent_journey` column is present, absent, or present-but-stale (JSON-first wins).
  - `validateParentJourney`: fixture covering accept + the four reject branches (empty is
    accepted/no-op; self; missing parent; parent-has-parent; journey-with-children).
- **TS behavioral (CI):**
  - `ensureMigratedOnOpen`: pending-TS-migration → lock+backup+apply; already-migrated →
    single `_migrations` query, no lock, no backup; missing-Python-migration → still refused.
  - Dual-write atomicity: `createJourney` writes column + JSON in one `withTransaction`;
    injected failure rolls back both (no half-write, no drift).
- **Real-legacy-DB-copy harness (redacted by default):** A1/A2/A5/A6 over
  `tmp/parity/demo-memory.db` generated by the existing demo generator — never the live DB.
  Emits probe labels, counts, hashes, pass/fail; no ids/content/embeddings.

## E2E Decision

**Required.** Migrate-on-open has no Python oracle, so it must be exercised end to end on a
real DB copy, not only on fixtures.

## Navigator Validation Route

1. Generate a demo DB and roll it back to pre-`017` state (harness helper):
   `uv run python ts/parity/generate_demo_memory_db.py --out tmp/parity/demo-memory.db`
   (then the harness strips `017` from `_migrations` + drops the column to simulate legacy).
2. Run a read command through the TS front door against the copy (e.g. `journeys`).

- **Expected observation:** a backup file appears in the owner-only backup dir (`0600`);
  the front-door log shows one `migrate_on_open` event naming `017` and the backup path;
  `journeys` output ordering is byte-identical to the pre-migration run.
- **Pass condition:** backup created + `_migrations` contains `017` + column populated from
  JSON + `journeys` output unchanged + second run performs no backup/migration.
- **Fail condition:** any ordering change; a backup on an already-migrated open; a second
  backup on re-open; journey/identity content appearing in the log; the column diverging
  from JSON after a Python parent write (expected under D1 — JSON-first read must still be
  correct, proving the column is a non-authoritative shadow).
