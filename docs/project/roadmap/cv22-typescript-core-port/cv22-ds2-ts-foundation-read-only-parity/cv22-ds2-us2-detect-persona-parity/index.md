# CV22.DS2.US2 — `detect-persona` Parity

**Status:** ✅ Done
**Type:** User Story
**Delivery Story:** CV22.DS2 — TS Foundation & Read-Only Command Parity
**Depends on:** [CV22.DS2.US1 `search` Command Parity](../cv22-ds2-us1-search-command-parity/index.md) (done), [CV22.DS2.TS3 Reusable Real-DB-Copy Parity Harness](../cv22-ds2-ts3-reusable-real-db-copy-parity-harness/index.md) (done)

## User Story

In order to keep extending the TypeScript core across the read-only command set with the same parity standard,
As a Mirror Mind operator whose ego routes personas by keyword,
I want the TS core to reproduce Python `detect-persona` routing decisions over the shared identity data,
So that persona routing can later move behind the TS front door without changing which persona a query activates.

## Outcome

The Python router `IdentityService.detect_persona` is ported into a durable, tested TS module (`ts/src/persona/detectPersona.ts`) and validated through the DS2 golden-corpus contract. The committed synthetic corpus drives the real Python oracle over probe queries that exercise every routing branch, and the TS verifier asserts exact behavioral parity (ordered persona keys, hit-count scores, and match type).

Unlike the hybrid ranker, `detect_persona` is pure and deterministic — no clock, no embeddings — so parity is exact behavioral equality rather than the ordered-id-with-float-drift metric `search` uses. The reusable real-DB-copy parity harness (DS2.TS3) is extended with a `detect-persona` probe path so the same redacted, copy-validated realism route now covers a second command family.

The story does not route any runtime command yet. It proves the routing behavior that DS3 will later place behind the Pi TS front door.

## Acceptance Behavior

```text
Given a synthetic golden corpus generated by the Python IdentityService.detect_persona oracle
And the corpus includes synthetic persona routing tables and probe queries exercising every branch
When the TS router scores each probe query against the routing tables
Then the returned persona keys, hit-count scores, and match types match the Python oracle exactly
```

```text
Given a database (portable demo or a copied real memory.db) that carries persona routing metadata
When the Python oracle and the TS router are run against the copy for derived and no-match probes
Then the ordered persona keys match for each probe
And no live production database is mutated
And evidence is redacted by default (labels, counts, hashes, pass/fail)
```

```text
Given the existing ts/ package constraints
When typecheck, lint, tests, and fixture determinism checks run
Then the TS detect-persona implementation stays inside the durable core structure
And no module outside the DB seam imports node:sqlite directly
```

## Scope

- Port `_normalize_routing_text` and `detect_persona` into durable TS modules under `ts/src/persona/`.
- Add a synthetic persona golden generator (`ts/parity/generate_persona_golden.py`) that seeds fully synthetic persona rows, drives the real Python oracle, and commits `personas`, `probes`, and `threshold`.
- Cover single-word token membership, multi-word substring matching, hyphen/underscore/punctuation normalization, score ties broken by ascending key, threshold boundary, and empty results.
- Extend the reusable real-DB-copy parity harness with a `detect-persona` probe path and keep evidence redacted by default.
- Extend the portable synthetic demo DB with synthetic persona rows so the persona parity route needs no private mirror.
- Add or update `node:test` coverage so synthetic parity and harness behavior run in CI.
- Add the persona golden to the CI determinism gate.

## Out Of Scope

- Routing the user-facing `detect-persona` command through TS (CV22.DS3).
- Journeys and memory listing parity (CV22.DS2.US3).
- Write commands, schema changes, or production database mutation.
- Any change to routing semantics, normalization behavior, or scoring. This is parity work.

## Validation

- Automated: `npm run typecheck`, `npm run lint`, `npm test`, and golden regeneration determinism for both goldens.
- Portable: generate the synthetic demo DB and run the reusable harness (search + detect-persona) against a copy.
- Safety: no real production database is committed or mutated; committed fixtures remain synthetic.
