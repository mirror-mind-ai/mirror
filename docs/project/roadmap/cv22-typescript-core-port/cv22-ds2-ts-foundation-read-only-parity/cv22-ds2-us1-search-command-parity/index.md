# CV22.DS2.US1 — `search` Command Parity

**Status:** ✅ Done
**Type:** User Story
**Delivery Story:** CV22.DS2 — TS Foundation & Read-Only Command Parity
**Depends on:** [CV22.DS2.TS2 Golden-Corpus Contract & Frozen-`now` Harness](../cv22-ds2-ts2-golden-corpus-contract/index.md) (done)

---

## User Story

In order to trust the TypeScript core as the first real read-only strangler slice,
As a Mirror Mind operator using memory search,
I want the TS core to reproduce Python `search` ranking behavior over the shared SQLite seam,
So that `search` can later be routed through the TS front door without changing observable results.

## Outcome

The DS1 spike ranker is promoted into durable `ts/` modules and validated through the DS2 golden-corpus contract. The committed synthetic corpus contains the ranker inputs needed to replay full hybrid scoring, and the TS verifier asserts ordered-id parity against the Python oracle.

Local implementation now includes `ts/src/search/ranker.ts`, exported from `ts/src/index.ts`, with synthetic ordered-id parity passing. Manual real-DB-copy parity also passed against a copied real database across five probes.

The story does not route any runtime command yet. It proves the core behavior that DS3 will later place behind the Pi TS front door.

## Acceptance Behavior

```text
Given a synthetic golden corpus generated by the Python `MemorySearch.search` oracle
And the corpus includes query embedding, frozen now, lexical surface, memory metadata, access signals, relevance, and embedding vectors
When the TS ranker scores and deduplicates the corpus
Then the ordered memory ids match the Python oracle exactly
```

```text
Given a real production database copied to a safe local parity path
When Python `search` and the TS ranker are run against the copy for selected probes
Then the ordered ids match for each probe
And no live production database is mutated
```

```text
Given the existing `ts/` package constraints
When typecheck, lint, tests, and fixture determinism checks run
Then the TS search parity implementation stays inside the durable core structure
And no module outside the DB seam imports `node:sqlite` directly
```

## Scope

- Promote the DS1 hybrid-search scoring behavior into durable TS modules under `ts/src/`.
- Extend the synthetic golden schema and generator with the remaining ranker inputs required for full replay: lexical/text surface, `use_count`, `relevance_score`, access count, `last_accessed_at`, created timestamp, embedding, and frozen query inputs.
- Implement TS scoring for semantic similarity, recency, reinforcement, manual relevance, ordinal lexical signal, stable score ordering, and MMR deduplication.
- Keep the success metric as ordered ids, not bit-identical scores.
- Add or update `node:test` coverage so synthetic parity runs in CI.
- Document and execute a manual real-DB-copy parity route.

## Out Of Scope

- Routing the user-facing `search` command through TS. That belongs to CV22.DS3.
- Fresh embedding generation for live user queries. External API behavior belongs to CV22.DS5.
- Porting `detect-persona`, journeys, or memory listing. Those remain CV22.DS2.US2 and CV22.DS2.US3.
- Write commands, schema changes, tokenizer changes, or production database mutation.
- Score improvements or ranking redesign. This is parity work.

## Validation

- Automated: `npm run typecheck`, `npm run lint`, `npm test`, and golden regeneration determinism.
- Manual: run parity against a copied real `memory.db` with selected probes and compare ordered ids between Python and TS.
- Safety: no real production database is committed or mutated; committed fixtures remain synthetic.
