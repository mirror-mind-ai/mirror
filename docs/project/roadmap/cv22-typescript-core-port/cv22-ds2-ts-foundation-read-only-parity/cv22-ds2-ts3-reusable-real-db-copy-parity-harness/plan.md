# Plan — CV22.DS2.TS3 Reusable Real-DB-Copy Parity Harness

## Objective

Turn the ad hoc `CV22.DS2.US1` real-DB-copy validation route into a small, safe, reusable harness for the rest of DS2. The harness should let a driver validate parity against a copied database with realistic data while preventing accidental leakage of private database content into committed fixtures, logs, or story evidence.

The portable demo source for validation is generated by the repository:

```bash
uv run python ts/parity/generate_demo_memory_db.py --out tmp/parity/demo-memory.db
```

The generated DB is synthetic and public in content, but the DB file itself remains a local generated artifact under ignored `tmp/` storage.

## Scope

- Promote the useful `tmp/` validation shape from US1 into a versioned harness under `ts/parity/` or another project-owned script location.
- Support an explicit `--source-db` input and copy it into ignored local storage before reading.
- Default generated artifacts to an ignored local path, for example `tmp/parity/`.
- Keep stdout redacted by default:
  - no memory content;
  - no conversation titles;
  - no raw memory ids;
  - no embeddings;
  - no full local user paths in commit-worthy evidence;
  - no real fixture JSON printed to stdout.
- Emit commit-safe evidence such as probe label, result count, ordered-id hash comparison, and pass/fail.
- Allow sensitive debug output only through an explicit opt-in flag with an intentionally clear name, if needed at all.
- Document how future DS2 parity stories should use the harness.
- Add tests or fixture-level checks proving the redacted output does not expose raw ids/content for representative input.

## Non-Goals

- Do not implement `detect-persona`, journeys, or memory listing parity.
- Do not require OpenRouter or fresh external embedding generation.
- Do not commit any copied real database, real-memory fixture, embedding dump, or personal data.
- Do not build a broad parity framework beyond the current DS2 handoff need.
- Do not mutate any source database.

## Acceptance Behavior

```text
Given a demo or real source memory.db path
When the harness runs with default settings
Then it copies the database to ignored local storage
And runs parity against the copy
And prints only redacted, commit-safe evidence
```

```text
Given the default output of the harness
When the output is inspected or pasted into validation docs
Then it contains probe labels, counts, hashes, and pass/fail status
And it does not contain raw memory ids, memory content, conversation titles, embeddings, or real fixture JSON
```

```text
Given a future DS2 command-parity driver
When they read the TS3 docs
Then they can run the harness against `/Users/alissonvale/.mirror-minds/_demos/lucas-vidal/memory.db` or another explicit local source DB without reconstructing the US1 temporary scripts
```

## Implementation Approach

1. Inspect the US1 temporary scripts and extract the reusable behavior into a versioned script.
2. Design the CLI contract around explicit inputs:
   - `--source-db <path>`;
   - optional `--work-dir <path>` defaulting to ignored local storage;
   - optional probe labels or built-in safe probe set for `search` parity validation.
3. Make the script copy the source database first and run only against the copy.
4. Redact stdout by default. Compare raw ids internally, but report hashes and counts externally.
5. Add a small automated test around the redaction/evidence renderer so accidental raw output becomes visible in CI.
6. Update TS3 `test-guide.md` with the demo DB route and expected redacted output.
7. Re-run the harness against the generated synthetic demo DB and record redacted evidence.

## Validation Route

Automated or focused local checks:

```bash
cd ts
npm run typecheck
npm run lint
npm test
```

Harness validation route from repository root:

```bash
uv run python ts/parity/generate_demo_memory_db.py --out tmp/parity/demo-memory.db
uv run python <harness command> \
  --source-db tmp/parity/demo-memory.db
```

Expected observation: the commands generate a synthetic demo DB locally, report redacted pass/fail evidence, and write copied/generated database artifacts only to ignored local storage.

Pass condition: checks pass, parity evidence is redacted by default, the demo DB source is not mutated, and git status does not show copied DBs or real-data artifacts.

Fail condition: output leaks raw ids/content/titles/embeddings, the source DB is mutated, generated real-data artifacts become tracked, or future drivers cannot run the documented route.

## E2E Decision

Runtime E2E is not required. This is a validation harness story, not a user-facing runtime route. The meaningful end-to-end route is running the harness against the Lucas Vidal demo DB copy and inspecting redacted evidence.

## Debt Policy

This story exists to pay the debt carried from US1. If the implementation reveals a broader parity framework is needed, capture that as future DS2/DS5 work rather than expanding TS3.
