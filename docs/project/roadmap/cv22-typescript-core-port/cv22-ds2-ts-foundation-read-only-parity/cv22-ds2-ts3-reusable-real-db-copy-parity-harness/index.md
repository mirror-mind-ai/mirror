# CV22.DS2.TS3 — Reusable Real-DB-Copy Parity Harness

**Status:** ✅ Done
**Type:** Technical Story
**Delivery Story:** CV22.DS2 — TS Foundation & Read-Only Command Parity
**Depends on:** [CV22.DS2.US1 `search` Command Parity](../cv22-ds2-us1-search-command-parity/index.md) (done)

## Technical Story

In order to pass the CV22.DS2 baton at a habitable plateau,
As the next driver of read-only parity work,
I want a reusable, safe, documented real-DB-copy parity harness,
So that `detect-persona`, journeys, and memory listing parity can reuse the same validation pattern without reconstructing it from local `tmp/` scripts or conversation memory.

## Outcome

The real-DB-copy validation route used during `CV22.DS2.US1` is now promoted toward a durable, versioned support harness instead of a local artifact. It remains safe by construction: it operates on an explicit source database copied into ignored local storage, does not mutate the source database, does not commit personal data, and emits Navigator-readable pass/fail evidence redacted by default.

Local implementation provides:

- `ts/parity/real_db_copy_parity.py` — creates the safe copy and fixture, then invokes TS verification;
- `ts/parity/real_db_copy_verify.ts` — verifies the generated fixture through the TS ranker;
- `ts/src/parity/realDbCopyParity.ts` — redacted evidence helpers and fixture evaluation;
- `ts/test/parity/realDbCopyParity.test.ts` — regression coverage for hash stability and redaction.

The selected validation source is a synthetic demo DB generated by the repository:

```bash
uv run python ts/parity/generate_demo_memory_db.py --out tmp/parity/demo-memory.db
```

The generated DB uses fictional public data and the real Mirror schema. The generated DB file remains in ignored local storage rather than being committed as a binary fixture.

## Acceptance Behavior

```text
Given a demo or real source memory.db path
When the harness runs with default settings
Then it copies the database to ignored local storage
And runs parity against the copy
And prints only redacted, commit-safe evidence
```

```text
Given the default output of the harness
When the output is inspected or pasted into validation docs
Then it contains probe labels, counts, hashes, and pass/fail status
And it does not contain raw memory ids, memory content, conversation titles, embeddings, or real fixture JSON
```

```text
Given a future DS2 command-parity driver
When they read the TS3 docs
Then they can generate a synthetic demo DB and run the harness without needing Alisson's private filesystem or reconstructing the US1 temporary scripts
```

## Scope

- Promote the useful parts of the `tmp/` real-DB-copy validation scripts from US1 into a supported, versioned location.
- Keep generated databases and generated real-data fixtures in ignored local storage.
- Provide documented commands for creating/running the validation route.
- Make the output concise enough to paste into story validation evidence.
- Preserve the database-seam safety discipline and redact stdout by default.

## Out Of Scope

- Implementing `detect-persona`, journeys, or memory listing parity.
- Fresh external embedding generation as a required path.
- Writing to a production database.
- Committing copied real databases, real-memory fixtures, embedding dumps, or personal data.
- Building a general-purpose test framework beyond the current CV22.DS2 need.

## Validation

- Automated: `npm run typecheck`, `npm run lint`, and `npm test` from `ts/`.
- Demo DB: run `uv run python ts/parity/generate_demo_memory_db.py --out tmp/parity/demo-memory.db` from repository root.
- Harness: run `uv run python ts/parity/real_db_copy_parity.py --source-db tmp/parity/demo-memory.db` from repository root.
- Safety: verify generated database artifacts remain ignored and absent from normal git status.
