# CV22.DS2.US3 — Journeys & Memory Listing Parity

**Status:** ✅ Done
**Type:** User Story
**Delivery Story:** CV22.DS2 — TS Foundation & Read-Only Command Parity
**Depends on:** [CV22.DS2.US2 `detect-persona` Parity](../cv22-ds2-us2-detect-persona-parity/index.md) (done)

---

## User Story

In order to close the read-only deterministic foundation of the TypeScript core,
As a Mirror Mind operator who lists journeys and browses recent memories,
I want the TS core to reproduce Python's journey-option ordering and recent-memory listing over the shared SQLite seam,
So that both reads can later move behind the TS front door without changing observable output, and CV22.DS2 can close.

## Outcome

The last two read-only deterministic surfaces are ported and validated, completing the second Baton 2 slice and the DS2 command set (`search`, `detect-persona`, journeys, memory listing).

The surface splits by nature, and the split drives the design:

- **Journeys — pure logic.** `JourneyService.list_journey_options` + `_sort_journey_options` is pure over `journey` identity rows: derive each journey's name (first content line), status (`**Status:**` regex), and parent (metadata JSON), then order roots-then-children with a stable `(status != 'active', name.lower())` comparator. Ported to `ts/src/journey/journeyOptions.ts` with a committed synthetic golden — CI-gated exact parity.
- **Memory listing — seam SQL read model.** `list_recent_memory_summaries` (dynamic `WHERE` + `ORDER BY created_at DESC LIMIT`) and `count_memories_by_type` are ported to `ts/src/memory/listing.ts` as a query builder + row→`MemorySummary` mapper. Following the database-seam philosophy, the ordering is pushed down to SQLite. Per the agreed **option B**, CI covers the query builder and mapper (no DB), and listing-order realism is proven in the reusable real-DB-copy harness against the copied DB.

The `--search` memory-listing path is out of scope: it reuses the US1 hybrid ranker. No runtime command is routed; that is CV22.DS3.

## Acceptance Behavior

```text
Given a synthetic golden generated by the Python JourneyService.list_journey_options oracle
And the corpus exercises roots/children, status ordering, name ordering, and an orphaned child
When the TS journey logic orders the same rows
Then the ordered journey ids and option DTOs match the Python oracle exactly
```

```text
Given a database carrying memories with varied types, layers, and journeys
When the Python read model and the TS listing read model run the same filters/limit against the copied DB through the seam
Then the ordered memory ids match for each probe
And count_by_type matches as a sorted set
And no live production database is mutated
```

```text
Given the existing ts/ package constraints
When typecheck, lint, tests, and fixture determinism checks run
Then only ts/src/db/database.ts imports node:sqlite in production source
And the query builder and mapper are unit-tested without a real database
```

## Scope

- Port journey option listing + hierarchical sort to `ts/src/journey/journeyOptions.ts` (pure) with `ts/parity/generate_journeys_golden.py` and a committed golden.
- Port `list_recent_memory_summaries` + `count_memories_by_type` to `ts/src/memory/listing.ts` over the `Database` seam (query builder + mapper).
- Extend the reusable real-DB-copy harness with `journeys` (pure replay) and `memory-listing` (seam read against the copied DB) probe families, redacted by default.
- Seed synthetic journey identity rows and diversified memory types/layers/journeys into the portable demo DB.
- Add the journeys golden to the CI determinism gate.
- `node:test` coverage: journey golden parity + units; listing query-builder/mapper units.

## Out Of Scope

- Routing journeys or memory listing through TS (CV22.DS3).
- The `--search` listing path (reuses the US1 ranker) and fresh embeddings (CV22.DS5).
- Write commands, schema changes, or production database mutation.
- Reproducing SQLite's sort in JS (ordering is pushed down to the seam).

## Validation

- Automated: `npm run typecheck`, `npm run lint`, `npm test`, and determinism for all three goldens.
- Portable: generate the demo DB and run the reusable harness (search + persona + journeys + listing) against a copy.
- Safety: no real production database is committed or mutated; committed fixtures remain synthetic.
