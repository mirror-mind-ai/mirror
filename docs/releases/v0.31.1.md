# v0.31.1 — Change Request Terminal Verbs

**Date:** 2026-07-20

## Highlights

- Adds `change-request park` / `reject` / `promote` (and the symmetric
  `refinement-story park`) as real Builder CLI verbs, closing the gap that
  previously made a Refinement Story holding a deferred, declined, or
  graduated Change Request unclosable through supported commands.
- A six-persona review (quality-assurance, database-architect,
  security-engineer, ai-engineer, devops-engineer, prompt-engineer) ran
  against the written plan before implementation began, and its findings
  were folded back into the plan rather than treated as commentary.
- Fixes a latent `_flow_event` bug where a parked/rejected/promoted Change
  Request was still reported as the *active* one.
- Fences Ariad transport markers (`<<<ARIAD:`/`<<<END:`) inside rendered
  Change Request free-text, closing a surface-injection path.
- Fixes the Refinement Story progress surface so a terminal Change Request
  is never shown as "remaining" while the same surface says no actionable
  Change Requests are left.
- CR023 (`kia-desktop`, `a6be8152`) — the production row that motivated this
  release — is Navigator-confirmed reconciled against the new verb's output
  shape via a read-only check. No migration was needed.
- Closes an open Ariad roadmap-parser investigation in documentation:
  CV20.DS13 had already fixed the underlying defects, but the ledger and
  investigation record still read "Open," misleading anyone auditing the
  Ariad runtime.
- Registers TD-002: the `backup` CLI path ignores `MEMORY_ENV`/
  `db_name_for_env()`, silently pointing at the wrong database file on a
  development or test environment.

## Where We Started

A Navigator working the `kia-desktop` journey needed to park a Change
Request — a deliberate build-1 defer, not a mistake and not a rejection.
The Refinement Workbench domain already modeled `parked` as a terminal
state, and the RS-close gate already depended on it. But no CLI verb
reached it: only `done` was wired, and `discard` *deletes* a captured
record — the wrong tool for a decision worth keeping. The only path was a
raw write to the production storage layer, bypassing the service entirely:
no validation of legal source state, no emitted surface, and a real risk of
leaving the runtime cursor dangling had the Change Request been active.
That is precisely the failure mode Ariad's "every transition emits a
verbatim surface" contract exists to prevent.

## What Changed

The fix was planned before it was built. A written plan went through six
professional lenses — quality-assurance, database-architect,
security-engineer, ai-engineer, devops-engineer, and prompt-engineer — and
each surfaced something the plan had not accounted for: parking a
`validated` Change Request would have silently overwritten its validation
evidence (database-architect), free-text `reason`/`notes`/`target` fields
render into a transport surface that could be forged with a fake marker
(security-engineer), and the Refinement Story progress surface would
report a newly-terminal Change Request as both "remaining" and "no
actionable Change Requests left" in the same output (quality-assurance).
All three were designed around and fixed in the implementation, not left
as follow-up debt.

`park`, `reject`, and `promote` now mirror the existing `complete_change_request`
vertical slice end to end: service functions with legal-state guards, a
single authoritative terminal-status constant, surface entries for each new
event, and CLI subparsers wired through the existing clean-error path. `park`
requires both a reason and a revisit trigger; `reject` keeps the record
(contrast `discard`, which deletes); `promote` records a delivery target
without mutating the roadmap. Terminal outcome notes are prepend-preserved
rather than overwritten, so a Change Request's prior lifecycle evidence
survives its terminal transition. `mm-build/SKILL.md` now names the real
commands and explicitly disambiguates `reject` from `discard` — the one
confusion in this set with an irreversible wrong answer.

Validation ran in the order the plan committed to: the full
park/reject/promote → review/coherence/close route confirmed by the
Navigator in Builder Mode, a four-phrase verb-routing probe confirming
natural language never mis-routes `reject` to `discard`, and — the one
production-touching step, run by the Navigator via a read-only script, not
the agent — CR023 itself, matched against the new verb's expected shape on
all four checks.

Separately, this release closes the loop on an earlier defect fix:
CV20.DS13 (Delivery Story grammar roadmap support) shipped the code fix for
a roadmap-parser investigation but never marked the investigation or its
dogfooding-ledger findings as resolved, leaving the written record actively
misleading. That record now matches reality. TD-002 — a real bug found
while preparing this release's own backup step, where `backup`'s CLI path
never consults `MEMORY_ENV` — is registered in the technical debt ledger,
not fixed here.

## Conscious Exclusions

- `promote` is deliberately minimal: it records a delivery target and
  reaches a terminal state, and nothing more. It does not create or mutate
  a roadmap item — Ariad does not act as an autonomous project manager.
- No automated verb-routing eval shipped. The manual routing probe plus the
  skill's reject-vs-discard disambiguation and a "never write storage
  directly" guardrail mitigate the immediate risk; an automated golden-set
  eval is registered on the roadmap Radar as a fast-follow.
- Refinement Workbench commands remain Pi-only in the skills — the Claude
  skill has no Change Request/Refinement Story commands, and Codex/Gemini
  carry no `mm-build` surface at all. Also registered on the Radar.
- TD-002 is registered, not fixed.
- CV20 remains in progress. This release closes one Delivery Story;
  CV20.DS7 (Release And Push Policies) and the remaining planned stories
  are unaffected.

## What We Learned

Reviewing a plan from six professional lenses before writing a line of code
found three real defects at zero implementation cost — the same class of
defect that, unreviewed, becomes exactly the kind of production incident
this release exists to close (CR023 itself). Separately, the DS13 cleanup
was a reminder that fixing code and closing the record of the problem are
two different acts: a harvest workflow that skips the second half leaves a
technically-solved defect looking permanently open to the next person who
reads the ledger.

## Next Horizon

CV20's primary arc continues toward CV20.DS7 (Release And Push Policies).
The two items registered on this release's Radar — an automated
verb-routing eval and Claude/Codex Refinement-skill parity — are the
visible next steps for Refinement Work specifically. CV9, CV21, and CV22
continue in parallel, unaffected by this release.
