# v0.30.0 — Windows Installer Distribution

**Date:** 2026-07-05

## Highlights

- Publishes the first official native Windows installer path for Mirror Mind + Pi.
- Keeps installer source under `installer/` and generated `.exe` binaries outside the repository tree.
- Adds release-tag CI support so the Windows Installer workflow runs for `v*` tags and can produce a tag-matched installer artifact.
- Clarifies the distribution model: new Windows users download the `.exe` from GitHub Releases; existing users update Mirror through `uv run python -m memory runtime update`.
- Preserves the installer as a bootstrapper with its own version in `installer/VERSION`, decoupled from the Mirror runtime version.

## Where We Started

The Windows installer had landed on `main`, but the user path was still unclear.
A new Windows user arriving at the GitHub repository would not see a committed
`.exe`, because generated binaries are intentionally excluded from the source
tree. The installer existed as source and as a GitHub Actions artifact, which is
acceptable for maintainers but confusing for public onboarding.

That exposed the missing release boundary: the installer needed to become a
published release asset, not just a CI artifact hidden under Actions.

## What Changed

This release makes the Windows installer distribution model explicit and
release-backed. The repository continues to version the installer source,
including `installer/mirror.iss`, PowerShell bootstrap/configure scripts, the
launcher, tests, assets, and documentation. The compiled `.exe` remains a build
artifact, not a committed file.

The Windows Installer workflow now also runs on version tags. That means a
promoted release can produce an installer artifact from the exact tagged source,
which can then be attached to the matching GitHub Release. This keeps the public
asset aligned with the release tag rather than with an arbitrary branch run.

For users, the intended path is now simple: go to the GitHub Releases page,
download `MirrorMind-Setup-<installer-version>.exe`, run it, provide a Mirror
user name and OpenRouter key, then open Mirror Mind and run `/login` in Pi for
the chat-model subscription. For existing users, the `.exe` is not the update
mechanism; the runtime updater remains the path for Mirror updates.

## Conscious Exclusions

- The installer `.exe` is still not committed to the repository tree. Generated binaries belong to CI artifacts and GitHub Releases.
- This release does not code-sign the Windows installer. SmartScreen warnings remain expected until signing is added.
- The installer version remains decoupled from the Mirror runtime version. `installer/VERSION` changes only when the installer itself changes.
- Existing Windows users do not need to reinstall for ordinary Mirror runtime updates.

## What We Learned

A working installer is not the same thing as a distributable installer. Source,
CI artifact, release asset, and user-facing download link are four different
surfaces. The public onboarding path only becomes coherent when the generated
binary is attached to a release users can actually find.

## Next Horizon

The next refinement is to make release publication smoother: keep generating the
installer from the tagged source, attach it to every relevant GitHub Release, and
later add code signing so Windows trust prompts match the intended public
onboarding experience.
