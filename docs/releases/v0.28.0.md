# v0.28.0 — Ariad Delivery Story Lifecycle

**Date:** 2026-06-16

## Highlights

- Adds Navigator flow-unit selection between `story_by_story` and `delivery_story`.
- Persists Delivery Story lifecycle state, child work packages, and aggregate checkpoint status.
- Adds aggregate Delivery Story Plan, approval, validation, debt review, coherence, and done runtime checkpoints.
- Materializes DS-level `plan.md`, `validation.md`, `review.md`, `coherence.md`, and `done.md` artifacts in canonical roadmap packages.
- Brings DS-level Ariad surfaces into the visual card grammar, including Builder resume, pull candidates, flow-unit selection, DS Plan, and DS closure.
- Hardens the sandbox reset fixture so `/Users/alissonvale/Code/sandbox-pet-store` reliably includes its operational `REFERENCE.md`.

## Where We Started

`v0.27.0` made Ariad operational for story-by-story Builder delivery, but Delivery Stories still behaved mostly as expansion containers. That preserved safety, yet it created friction for cohesive work where the Navigator wants to validate the whole Delivery Story rather than approve and close every child User or Technical Story separately.

Dogfooding made the missing layer visible. A Delivery Story needed to become a selectable Navigator-facing flow unit without erasing child story traceability. The runtime also needed to stop treating DS-level checkpoints as transient console output: Plan and closure evidence had to land in the same canonical package where the roadmap says the DS lives.

## What Changed

Builder now supports two Ariad Navigator flow units:

```text
story_by_story  -> child User/Technical Stories are Navigator-facing lifecycle units
delivery_story  -> parent Delivery Story is the Navigator-facing lifecycle unit
```

In `delivery_story` flow, child stories remain visible as Driver work packages and evidence units, while the Navigator approves the aggregate DS Plan and validates the aggregate DS outcome. The runtime persists the chosen flow unit, child work items, and aggregate checkpoint status in the delivery cursor so Builder can resume the DS lifecycle coherently.

The DS Plan and closure lifecycle is now materialized. `plan-delivery-story` writes `plan.md`; DS validation, review, coherence, and done write `validation.md`, `review.md`, `coherence.md`, and `done.md` in the canonical Delivery Story package. Path resolution prefers existing roadmap packages so the runtime does not create synthetic fallback paths when canonical docs already exist.

The release also tightens Navigator experience. Previously some newer Ariad surfaces were valid transport blocks but rendered as plain structured text. They now use the same visual card grammar as the core lifecycle surfaces, including `BUILDER_RESUME`, `PULL_CANDIDATES`, `NAVIGATOR_FLOW_UNIT`, `DELIVERY_STORY_PLAN_CHECKPOINT`, and `DELIVERY_STORY_CLOSURE_CHECKPOINT`.

## Conscious Exclusions

- No automated push or release policy is introduced here. CV20.DS6 owns commit/push/release governance.
- No durable debt ledger or pay-now Refactor loop is introduced here. CV20.DS7 owns that layer.
- No project-local Ariad preference override file is introduced here. CV20.DS8 owns method preferences and effective-policy resolution.
- Delivery Story flow does not erase child story evidence; it changes the Navigator-facing lifecycle unit only.

## What We Learned

The important distinction is now explicit: User/Technical Stories can be Driver work packages while the Delivery Story is the Navigator's validation surface. That lets Builder preserve engineering traceability without forcing the Navigator to experience every internal unit as a separate product checkpoint.

The dogfood session also exposed a visual-contract lesson. Marked Ariad blocks are necessary but not sufficient; a Navigator-facing surface should also follow the visual grammar. Several surfaces were corrected during validation, which suggests a future refactor toward a shared Ariad card renderer instead of module-local formatting helpers.

## Next Horizon

The next release boundary is governance. CV20.DS6 should make commit, push, and release intent explicit in Builder Mode, separating local history from remote publication and stable promotion. After that, CV20.DS7 can make technical debt decisions durable through a ledger and Refactor loop.
