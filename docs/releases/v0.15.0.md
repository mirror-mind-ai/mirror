# v0.15.0 — Web Operations Runner

**Date:** 2026-05-26

## Highlights

- Added a server-owned allowlist catalog for web operations.
- Added runnable operations for runtime health, database backup, and conversation journey repair.
- Added local audit persistence in `operation_runs` and a recent-runs API.
- Added the visible Operations page with compact operation rows, operation detail panels, readable result cards, and recent audit evidence.
- Kept execution synchronous-first, with no arbitrary shell command, raw SQL, restore, delete, update, background job, streaming, or cancellation surface.

## Where We Started

Mirror Web had become useful for visibility: Workspace, Identity, configuration, conversations, and attachments were inspectable without going to SQLite or the terminal. But operational care still lived mostly in CLI commands. Runtime health, database backup, and conversation journey repair were real maintenance needs, but asking users to run the right command with the right flags kept the web surface contemplative rather than operational.

The risk was symmetrical. Moving operations into the browser could make Mirror easier to care for, but it could also accidentally become a remote command runner. The release had to create action without giving the web raw power over the machine.

## What Changed

The first layer was the operation contract. Mirror Web now exposes a catalog of server-owned operations with ids, descriptions, risk levels, dry-run behavior, execution state, and parameter declarations. The browser does not invent operations or submit command strings; it can only request operations that the server has declared.

The second layer was bounded execution. `runtime-health` runs read-only runtime inspection and returns structured status data. `database-backup` creates and optionally verifies a local backup using the existing backup path for the selected Mirror home. `conversation-journey-repair` previews high-confidence journey association repairs by default and only applies them when `dryRun` is explicitly false, creating a backup first.

The third layer was evidence. Operation runs are recorded locally in the selected Mirror database through the new `operation_runs` table. Successful and failed known-operation requests receive a `runId`, and recent runs can be listed through the web API. The audit record stores bounded parameters, status, outcome, timestamps, summaries, result evidence, and errors.

The final layer was the surface. The Operations page shows operations as compact rows, keeps configuration inside an operation detail panel, renders operation-specific result cards, keeps raw JSON collapsed as evidence, and presents recent audit history as a simple list. Runtime health now explains what needs attention instead of only reporting `attention needed`.

## Conscious Exclusions

This release does not introduce background jobs, streaming events, cancellation, retry, scheduling, backup restore, backup download, backup deletion, runtime update execution, extension installation, arbitrary filesystem paths, arbitrary SQL, or arbitrary shell commands.

Those exclusions are part of the safety boundary. v0.15 proves that Mirror Web can act through small, allowlisted, auditable operations before it learns to coordinate long-running work.

## What We Learned

The Operations surface clarified that audit evidence is not the same thing as a user interface. Raw JSON was useful while validating the contract, but the release only became coherent when results were rendered as operation-specific cards and raw payloads moved into collapsible evidence.

The work also clarified the next architectural threshold. Synchronous execution is enough for small maintenance operations, but it should not be stretched into long-running work. The future job model should be introduced deliberately, not smuggled in through larger POST requests.

## Next Horizon

The visible horizon is an operation job model with polling, then streaming events and cancellation for operations that can take longer or need progress feedback. That future layer should build on the contracts created here: operation ids, declared parameters, local audit records, and explicit user approval.
