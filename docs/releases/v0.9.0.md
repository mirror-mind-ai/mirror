# v0.9.0 — Self-Update Done

**Date:** 2026-05-23

## Highlights

- Stable update surfaces now prefer release language when local refs contain
  newer release notes.
- Release notes are available across Pi, Gemini/Codex shared skills, and Claude
  Code.
- `runtime release-doctor` provides a read-only promotion readiness checklist.
- `runtime release-promote` provides a controlled stable promotion path with
  dry-run, local promotion, and explicit push.
- Fresh-user stable update smoke now has an isolated procedure and validated the
  current stable no-op path without touching production data.

## Where We Started

`v0.8.0` made self-update safe enough to exist: status, diagnosis, backup,
dry-run, execution, updater repair, release notes, and stable/main channels were
in place. But the release layer around that mechanism was still thin. Stable
updates could still look like commit movement. Release notes existed, but runtime
skill access was uneven. Promotion relied on human git steps. And the user-shaped
smoke test had not yet proven what a fresh clone would experience.

The remaining question was not whether Mirror could update in principle. It was
whether the release path around update was coherent enough to be repeated.

## What Changed

Release-aware update notices now connect stable updates with narrative release
notes when the local upstream ref contains newer release metadata. `runtime
update --check` remains conservative and non-mutating; it can report a remote
commit without pretending to have fetched release-note details. Dry-run and
successful stable updates can speak in terms of the release being installed.

Release notes became available across runtime skill surfaces. Pi keeps
`/mm-release-notes`, Gemini and Codex receive the shared `.agents` skill, and
Claude Code has `/mm:release-notes`. The skill contract stays thin: call
`python -m memory runtime release-notes` and show the output verbatim unless the
user asks for a summary.

Release promotion became inspectable and then executable. `runtime
release-doctor` checks the repository, clean tree, package version, release note,
release index, tag state, and stable ref relationship without mutation. `runtime
release-promote` uses that doctor as a gate, creates or reuses the release tag,
creates or fast-forwards the local stable branch only when safe, and publishes
only when `--push` is explicit.

The fresh-user smoke story then tested the stable update path in a temporary
clone and temporary Mirror home. The smoke validated the current published stable
state as a no-op update and revealed an important isolation rule for historical
stable versions: set both `MIRROR_USER=` and `MIRROR_HOME=<temp-home>`, because
`MEMORY_DIR` alone does not prevent older code from loading `MIRROR_USER` from
`.env`.

## Conscious Exclusions

- This release does not close CV9 or claim Mirror Mind 1.0 readiness.
- The fresh-user smoke could not prove a `v0.8.0 -> v0.9.0` hop before this
  release was prepared and promoted; that hop is the post-promotion validation
  target.
- The release promotion path does not query GitHub Actions, wait for CI, force
  push, rewrite tags, bump versions, or write release notes.
- Production update validation remains a controlled operational step after
  stable promotion.

## What We Learned

A self-updating runtime needs a release vocabulary, not only update mechanics.
Users should see releases, not just commits. Maintainers should see readiness,
not just git commands. And smoke tests should be shaped like real users, not like
the developer's machine.

The visualization experiment also clarified how Ariad work should be shown:
Bird's-Eye maps locate work, Transition Views show movement, Release Intent
shows the version arc being built, and checkpoint views should show only the
components useful for that stage.

## Next Horizon

After `v0.9.0` is promoted to stable, the next validation is the real fresh-user
update hop from `v0.8.0` to `v0.9.0`. Once that passes, CV9.E3's Self-Update Done
track can close and the project can return to the remaining CV9 readiness work:
boundary hardening and stabilization before the eventual Mirror Mind 1.0
boundary.
